# BrowseCraft P3-9.6 Completeness Regression - Run 1

Date: 2026-07-04

## Scope

- Audit whether P3-9 Source Config Neutralization reached the intended architecture boundary.
- Verify that `Source` is configuration-first, while `SiteRule` remains only a rule-source configuration and migration-compatible access path.
- Check that Library and Reader no longer assume every source has `source.rule`.
- Check that rule-only execution use cases are placed under `RuleSourceRuntime` boundaries.
- Run targeted App and Core regression tests.

## Architecture Audit

### Confirmed

- `Source` now stores `configuration: SourceConfiguration` as the primary field.
- `source.rule` is a compatibility accessor and fails only for non-rule-backed sources.
- `SourceRecord.domainModel()` can return runtime-neutral `.rule`, `.rss`, and `.plugin` `Source` values through `SourceConfiguration`.
- `LibraryViewModel` no longer reads `source.rule` directly; rule-specific presentation is behind `ResolveLibrarySourcePresentationUseCase`.
- `ReaderViewModel` and `ChapterListViewModel` no longer read `source.rule` directly; rule-specific presentation is behind `ResolveReaderSourcePresentationUseCase`.
- Rule-only search and reader execution implementations are under `Application/Runtime/Rule`.
- `SearchSourceUseCase`, `RuleSourceLoadChaptersUseCase`, and `RuleSourceLoadReaderChapterUseCase` are clearly RuleSourceRuntime internals.
- App-level `LoadChaptersUseCase` and `LoadReaderChapterUseCase` remain as Reader feature facades.
- No `Bridge`, `Adapter`, or `unsupportedPersistedSourceKind` naming drift was found.
- No RSS, plugin, or video fields were added to `SiteRule`.
- Core remains independent from App-only implementation details.

### Accepted Transitional Boundaries

- `RefreshSourceUseCase` remains in `Application/UseCases` because `SourcesViewModel` still calls it directly.
- `RuleDebugUseCases`, `RuleManagementUseCases`, and `RulePackageUseCases` still read `source.rule`; these are rule-editing/debug/package boundaries and are expected to remain rule-specific.
- `RuleParsingService`, `URLResolvingService`, `SwiftSoupRuleParser`, and `SwiftSoupRuleCandidateAnalyzer` still read `source.rule`; these are rule-parser boundaries and should move only if the parser API is redesigned later.
- `RuleDetailView` still reads `source.rule`; this is the rule editor/detail UI and is outside the P3-9 neutral runtime goal.
- `Application/Adapters/` remains as an empty physical directory and can be removed in a later hygiene task if desired.
